# 前端資安檢查報告 (Investment Assistant Platform)

## Executive Summary
本次審查基於 OWASP Top 10:2025 框架。專案在認證機制與基礎架構上具備良好品質（如 HttpOnly Cookie），但在**跨站腳本攻擊 (XSS)**、**跨站請求偽造 (CSRF)** 與**存取控制保護**上仍存在顯著優化空間。

### Top 5 核心風險：
1.  **XSS 潛在風險 (High)**：AI 生成內容直接透過 Markdown 渲染，未經過濾可能導致惡意腳本執行。
2.  **CSRF 防禦缺失 (High)**：在使用 Cookie 認證的情況下，未實作 CSRF Token 機制。
3.  **敏感 ID 洩漏與越權風險 (Medium)**：自增 ID (BIGINT) 易被推測，前端直接暴露且未見防護。
4.  **敏感日誌洩漏 (Low)**：全域錯誤處理器將完整 Error 對象印至 Console。
5.  **缺乏內容安全策略 (CSP) (Low)**：`index.html` 未設定 CSP Header。

---

## 依 A01~A10 條列檢查結果

### A01:2025-Broken Access Control (失效的存取控制)
*   **判定**：Potential Risk
*   **證據**：`docs/01_專案架構書.md` 第 35 行提及內部主鍵使用 `BIGINT` 自增 ID；`src/api/chat.api.ts` 與 `src/hooks/useImportFlow.ts` 中頻繁傳遞這些 ID。
*   **攻擊路徑**：攻擊者可透過遞增 `portfolioId` 或 `conversationId` 的方式嘗試存取不屬於自己的投資組合或聊天紀錄。
*   **風險等級**：High (若後端未做 Ownership Check) / Medium (資訊洩漏)
*   **修正方案**：
    1.  後端應在每個 API 實作用戶所有權檢查（Ownership Filter）。
    2.  前端與後端溝通建議改用 **UUID v4** 作為對外標識符，避免 ID 被列舉。
*   **驗收方式**：使用不同帳號登入，手動修改 URL 或 API Payload 中的 ID，確認是否能抓取到他人的數據。

### A03:2025-Injection (注入攻擊 - 含 XSS)
*   **判定**：Potential Risk
*   **證據**：`src/components/ai/FloatingAiAssistant.tsx` 與 `src/components/ai/AiAnalysisModal.tsx` 使用 `react-markdown` 渲染 AI 回覆。
*   **攻擊路徑**：若 AI 被注入惡意指令或後端資料來源遭污染，產出的 Markdown 包含 `<img src=x onerror=alert(1)>` 或 `javascript:` 連結，將導致 XSS。
*   **風險等級**：High
*   **修正方案**：引入 `rehype-sanitize` 插件。
    ```bash
    npm install rehype-sanitize
    ```
    ```tsx
    import rehypeSanitize from 'rehype-sanitize'
    <ReactMarkdown rehypePlugins={[rehypeSanitize]}>{content}</ReactMarkdown>
    ```
*   **驗收方式**：輸入包含 HTML 標籤與惡意事件的內容給 AI，檢查渲染後標籤是否被移除或跳脫。

### A05:2025-Security Misconfiguration (安全性設定錯誤)
*   **判定**：Potential Risk
*   **證據**：`apps/frontend/index.html`
*   **攻擊路徑**：缺乏 CSP (Content Security Policy) 保護，無法防止未經授權的外部腳本載入或資源連線。
*   **風險等級**：Low
*   **修正方案**：在 `index.html` 或 Nginx 增加 CSP Meta 標籤，僅允許白名單網域（如 API Domain, Sentry, Google Auth）。
*   **驗收方式**：使用瀏覽器開發者工具檢查 Header 是否包含 `Content-Security-Policy`。

### A07:2025-Identification and Authentication Failures (認證及識別失效)
*   **判定**：Potential Risk
*   **證據**：`src/utils/http.ts` 配置 `withCredentials: true`。
*   **攻擊路徑**：專案使用 Cookie 作為認證載體，若後端未校驗 `Origin` 或 `Referer`，且前端未實作 CSRF Token，攻擊者可誘導用戶點擊惡意連結進行偽造請求。
*   **風險等級**：High
*   **修正方案**：
    1.  **後端**：啟用 CSRF 防護（如 Spring Security 的 `csrfTokenRepository`）。
    2.  **前端**：在 Axios 攔截器中讀取 `XSRF-TOKEN` Cookie 並放入 Header `X-XSRF-TOKEN`。
*   **驗收方式**：使用工具（如 Postman）在不帶 CSRF Token 的情況下嘗試對 POST API 發送請求，應被攔截。

### A09:2025-Security Logging and Monitoring Failures (安全性記錄及監控失效)
*   **判定**：Potential Risk
*   **證據**：`src/hooks/useApiError.ts` 第 9 行 `console.error('API Error...', error)`。
*   **攻擊路徑**：若錯誤對象包含完整的 Axios Response，可能無意間洩漏了 User Email 或部分 Token 資訊至客戶端日誌中。
*   **風險等級**：Low
*   **修正方案**：在生產環境中移除 `console.log/error`，或使用 Sentry 提供的 `beforeSend` 回呼函數進行數據脫敏（Scrubbing）。
*   **驗收方式**：在生產環境（npm run build）後檢查瀏覽器控制台是否仍有敏感資訊流出。
