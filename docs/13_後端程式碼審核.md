# 後端程式碼審核報告

> 產生時間：2026-02-06
> 審核對象：Backend (Spring Boot)

本報告針對現有後端程式碼進行 **高併發 (High Concurrency)**、**低耦合 (Low Coupling)**、**程式碼品質 (Code Quality/Bloat)** 與 **潛在 Bug** 的深度審核。

## 1. 高併發設計 (High Concurrency)

### ✅ 優點
1.  **Single Flight 機制完善**：
    -   在 `StockQuoteService` 與 `WarrantQuoteService` 中，使用了 `ConcurrentHashMap<String, CompletableFuture<Object>>` 實作 Single Flight 模式。
    -   **效果**：當大量請求同時查詢同一檔股票（e.g., `2330`）時，只會觸發一次對外的 API 呼叫，其餘請求等待同一個 Future 完成。這極大降低了對 Fugle/Alpaca 等外部供應商的壓力，是高併發系統的標準解法。
2.  **非同步處理 (Async Processing)**：
    -   `AiConversationController` 使用 `CompletableFuture.runAsync` 配合 `SseEmitter` 處理 AI 回覆。
    -   `AiController` 同樣使用 Async 處理，避免長時間佔用 Servlet Container 的 Thread Pool (Tomcat threads)，提升系統吞吐量。
3.  **Redis Streams 佇列**：
    -   OCR 任務採用 Redis Streams (`OcrQueueService`) 進行非同步解耦。這允許 `app-ocr` 快速接收請求並回傳，而由背景 Worker 慢慢消化重度運算，避免前端 timeout。
4.  **Rate Limiter**：
    -   `SimpleRateLimiter` 實作了基於 `ConcurrentHashMap` 的執行緒安全計數器，雖是單機版但已具備基礎保護能力。

### ⚠️ 改進建議
1.  **OcrQueueService 的鎖競爭**：
    -   `readBatch` 與 `ensureGroup` 中使用了 `synchronized(this)`。雖然僅用於初始化 Consumer Group，但在極端併發重啟或網路抖動導致頻繁重連時，可能成為瓶頸。建議由外部 Script 或系統啟動時一次性建立 Group，移除 Runtime 的鎖判定。

---

## 2. 低耦合設計 (Low Coupling)

### ✅ 優點
1.  **Maven Multi-Module 結構清晰**：
    -   `app-stocks` 的 `pom.xml` 顯示它僅依賴 `app-common` 與 `app-persistence`，完全不知道 `app-api` 的存在。這符合 Clean Architecture 的單向依賴原則。
    -   `app-api` 作為組裝層 (Controller)，依賴各業務 Module (`app-stocks`, `app-ocr`)，職責分明。
2.  **Controller 與 Service 分離**：
    -   `StockController` 僅負責參數驗證與 HTTP 回應封裝，複雜邏輯皆下放到 `StockQuoteService` 與 `InstrumentService`。
3.  **依賴注入 (DI)**：
    -   廣泛使用 Lombok `@RequiredArgsConstructor` 進行 Constructor Injection，利於單元測試 (Unit Test) 與 Mocking，避免了 Field Injection (`@Autowired`) 的隱性依賴問題。

---

## 3. 程式碼品質與臃腫度 (Code Bloat)

### 📊 檔案分析
-   **OcrService.java (~900 lines)**：
    -   **狀況**：目前偏大。它同時包含了「OCR Job 流程控制」、「DTO 轉換」、「資料正規化 (Normalization)確認邏輯」與「交易檢查」。
    -   **風險**：維護成本較高，閱讀時需頻繁跳轉。
    -   **建議**：可將 `toDraftEntity`、`normalizeQuantity` 等約 200 行的轉換邏輯抽出為 `OcrRequestMapper` 或 `OcrDataNormalizer`；將重複交易檢查抽出為 `TradeValidator`。

-   **StockQuoteService.java (~200 lines)**：
    -   **狀況**：精簡且專注。專注於快取與資料源調度，無多餘邏輯。

### ⚠️ 潛在邏輯問題 (Bugs & Fragility)
1.  **OcrService 的 saveDrafts 複雜度**：
    -   `saveDrafts` 方法中混合了 Hash 計算、重複檢查與警告標記。若未來新增更多檢查規則（如漲跌幅限制），此方法會迅速膨脹。建議引入 `RuleEngine` 或 `Validator Chain` 模式。
2.  **OcrQueueService 的 ensureGroup 冗餘邏輯**：
    -   `ensureGroup` 中使用了 try-catch 包裹 `createGroup`，若失敗又嘗試 add record 再 create。這種「試誤法」邏輯較脆弱，建議明確檢查 key 存在性或由部署腳本保證 Stream 存在。

---

## 4. 總結

**後端程式碼品質總體評分為 A-。**

-   **架構面**：Multi-Module 分層落實良好，未見循環依賴或跨層汙染。
-   **效能面**：關鍵路徑 (Quote/AI) 皆已實作快取與非同步與 Single Flight，具備承載高併發的體質。
-   **維護性**：大部份 Service 都在 300 行以內，唯獨 `OcrService` 承載過多業務細節，是未來重構的首要目標。

**下一步行動建議**：
1.  **重構 OcrService**：將資料轉換與驗證邏輯抽出。
2.  **增強 Rate Limiter**：目前的 In-Memory Limiter 無法跨實例 (Scale Out)，若未來部署多台 Backend，需改用 Redis Based Rate Limiter (如 Bucket4j + Redis)。
