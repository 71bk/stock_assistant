# 可觀測性與日誌

> 狀態：已實作基礎日誌 (2026-02-06)，進階監控待實作

## Log 格式

系統使用 **SLF4J + Logback** 進行日誌記錄。

- **Console Output**: 開發環境預設輸出至 Console，方便除錯。
- **File Output**: 生產環境建議設定 rolling file appender (透過 `logging.file.name` 或 logback 設定)。
- **Log Level**:
  - `ERROR`: 系統發生預期外的錯誤 (500)，包含 Stack Trace。
  - `WARN`: 業務警示 (如 404 Not Found, 400 Bad Request, 或第三方 API 失敗)。
  - `INFO`: 關鍵流程節點 (如 Application Started, Cron Job Started, OCR Job Completed)。
  - `DEBUG`: 開發時期的細節資訊 (預設關閉)。

## TraceId/MDC

為了追蹤跨請求的日誌，建議於 `Filter` 層級實作 **MDC (Mapped Diagnostic Context)**：

1. 每個 Request 進入時，生成一組唯一 `traceId` (若 Header `X-Trace-Id` 已存在則沿用)。
2. 將 `traceId` 放入 `MDC`。
3. Log Pattern 設定 `[%X{traceId}]`，確保所有 Log 都帶有該 ID。
4. Request 結束時清除 MDC。

*(目前專案中尚未詳盡實作完整的 TraceId Filter，建議作為下一階段優化重點)*

## Metrics

系統基於 **Spring Boot Actuator** + **Micrometer**。

- **Endpoint**: `/actuator/metrics`, `/actuator/prometheus` (需開啟)。
- **關鍵指標**:
  - JVM (Memory, GC, Threads)
  - Logback (Log 數量)
  - HTTP Server (Requests, Latency, Status Codes)
  - Connection Pool (DB, Redis)

## Error Tracking

所有未捕捉的異常均由 `GlobalExceptionHandler` 統一處理：
- **BusinessException**: 記錄錯誤碼與訊息 (WARN/INFO)。
- **Unexpected Exception**: 記錄完整 Stack Trace (ERROR)，需重點監控。

## 告警策略 (建議)

- **P0 (Critical)**:
  - 系統無法啟動
  - DB/Redis 連線失敗
  - `GlobalExceptionHandler` 捕獲的 500 錯誤率暴增
- **P1 (High)**:
  - OCR Job 失敗率過高
  - 第三方行情 API (Fugle/Alpaca) 連續失敗
  - Disk Space Low
