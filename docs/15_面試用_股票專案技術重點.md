# 面試用：股票投資助理專案技術重點

> 專案定位：股票投資助理平台  
> 核心能力：行情查詢、投資組合管理、OCR 匯入交易、AI 對話分析、RAG 知識檢索  
> 面試目標：能清楚說明「用了什麼技術、為什麼這樣選、相比替代方案的優勢、在專案中解決了什麼問題」。

---

## 1. 專案總覽

這個專案是一個股票投資助理平台，目標是讓使用者可以管理自己的持股、交易紀錄、查看台股與美股行情，並透過 OCR 匯入券商對帳單，再用 AI 進行投資組合分析與文件問答。

整體架構採用三個主要應用：

| 區塊 | 技術 | 職責 |
|---|---|---|
| Frontend | React 19 + TypeScript + Vite + Ant Design | 使用者介面、圖表、交易/OCR/AI 操作流程 |
| Backend | Java 21 + Spring Boot 4 + Maven Multi-Module | API、認證、交易/持倉、行情代理、OCR/RAG 協調 |
| AI Worker | Python + FastAPI | OCR、文件解析、Embedding、RAG ingestion/query |
| Database | PostgreSQL + JSONB + pgvector | 關聯資料、半結構化資料、向量檢索 |
| Cache / Queue | Redis | 行情快取、Refresh session、OCR queue、rate limit |
| Infra | Docker Compose | 本機與部署環境服務編排 |

面試可以這樣講：

> 我把系統切成前端、Java 後端、Python AI Worker 三個部分。Java 後端負責穩定的業務交易與安全控管，Python Worker 負責 OCR、文件解析與向量處理，因為這些任務更適合 Python 的 AI/資料處理生態。資料庫選 PostgreSQL，搭配 JSONB 與 pgvector，讓交易資料、OCR 結果與 RAG 向量可以在同一套資料庫中維護一致性。

---

## 2. 前端技術選型

### React 19 + TypeScript

| 選型 | 為什麼使用 |
|---|---|
| React | 適合 SPA，元件化清楚，生態成熟，方便切分 Dashboard、Portfolio、OCR Review、Chat 等頁面 |
| TypeScript | 股票、交易、金額、日期、API response 都需要明確型別，能降低前後端欄位不一致造成的錯誤 |

為什麼不用 Vue：

- Vue 也能完成這類後台系統，但本專案需要大量組合式 UI、狀態管理、表單、圖表與 AI 串流互動，React 生態在這些場景下選擇更多。
- React 搭配 TypeScript、React Query、Zustand 的模式很適合複雜資料流。

為什麼不用 Next.js：

- 這個系統主要是登入後使用的投資管理工具，不是 SEO 導向網站。
- 大多數頁面都需要登入與即時 API 資料，SSR 的收益有限。
- Vite SPA 開發速度快，部署也簡單。

面試講法：

> 前端選 React + TypeScript，是因為這是一個登入後的資料密集型 SPA，需要管理很多交易、持倉、行情與 AI 串流狀態。TypeScript 能讓 API contract、金額欄位與 enum 更安全。沒有選 Next.js，主要是因為這不是 SEO 型網站，SSR 的收益不高，用 Vite SPA 更直接。

### Vite

| 選型 | 為什麼使用 |
|---|---|
| Vite | 開發啟動快、HMR 快、設定簡潔，適合 React SPA |

為什麼不用 Webpack：

- Webpack 彈性高，但設定成本較高。
- Vite 對現代瀏覽器與 TypeScript 專案更輕量，開發體驗更好。

面試講法：

> Vite 的優勢是開發啟動與 HMR 很快，對前端功能快速迭代很重要。這個專案不是需要高度客製 bundling 的老系統，所以不用 Webpack 可以降低設定成本。

### Ant Design

| 選型 | 為什麼使用 |
|---|---|
| Ant Design | 表格、表單、Modal、Layout、日期選擇器等後台系統元件完整 |

為什麼不用 MUI：

- MUI 也成熟，但 Ant Design 對後台管理系統、資料表格與表單工作流的預設體驗更貼近本專案。
- 本專案有交易列表、OCR Review、Admin Dashboard、Knowledge Base 等偏企業工具的畫面，AntD 可以降低 UI 基礎建設成本。

為什麼不用純 Tailwind：

- Tailwind 很適合自訂設計，但本專案需要大量穩定的表單、表格與互動元件。
- 若全部自己刻，會增加可用性與維護成本。

面試講法：

> Ant Design 讓我可以把重點放在交易流程、OCR Review、AI 對話與資料整合，而不是重複打造表格、表單、Modal 這些基礎元件。對這類資料密集型後台工具來說，它比純 CSS 或 Tailwind 更省成本。

### Zustand + React Query

| 技術 | 職責 |
|---|---|
| Zustand | 管理前端本地 UI 狀態，例如 auth、chat、portfolio、stocks store |
| React Query | 管理 server state，例如 API 查詢、快取、重新抓取 |

為什麼不用 Redux：

- Redux 適合大型全域狀態，但樣板碼較多。
- 本專案多數資料是 server state，應該交給 React Query；真正的 client state 用 Zustand 即可。

為什麼不用只靠 Context：

- Context 適合低頻全域設定，但不適合大量狀態更新。
- 股票行情、聊天、匯入流程都會有較頻繁狀態變化，Zustand 比 Context 更輕。

面試講法：

> 我把 server state 跟 client state 分開。API 資料交給 React Query 管快取與重抓，本地 UI 狀態交給 Zustand。這樣比 Redux 輕，也比只用 Context 更適合頻繁變動的狀態。

## 3. 後端技術選型

### Java 21 + Spring Boot 4

| 選型 | 為什麼使用 |
|---|---|
| Java 21 | 長期支援版本，效能穩定，適合金融與交易資料處理 |
| Spring Boot | 生態成熟，支援 Security、JPA、Redis、Actuator、Validation、Testing |

為什麼不用 Node.js：

- Node.js 適合 I/O 密集與快速開發，但金融帳務、交易一致性、複雜 domain service 在 Java/Spring 生態下更穩定。
- Spring 對交易邊界、認證授權、資料存取、測試與可觀測性支援完整。

為什麼不用 Python 做整個後端：

- Python 很適合 AI/OCR/資料處理，但主後端涉及認證、交易一致性、API 合約與長期維護。
- Java 在型別、安全、併發、企業級架構上更適合做核心業務後端。

面試講法：

> 我把核心後端放在 Java/Spring Boot，是因為交易、持倉、認證、權限與資料一致性都需要穩定的框架支持。Python 沒有不用，而是放在 AI Worker，讓它處理 OCR 和 RAG 這些更適合 Python 生態的任務。

### Maven Multi-Module

後端模組：

| Module | 職責 |
|---|---|
| app-bootstrap | 啟動與組裝 |
| app-api | Controller、DTO/VO、API entry、SSE |
| app-auth | OAuth、JWT、Cookie、Redis refresh session、Admin login |
| app-portfolio | 投資組合、交易、持倉、估值 |
| app-stocks | 行情代理、商品主檔、Fugle/Alpaca/TPEx/TWSE adapter |
| app-ocr | OCR job、queue、draft、validator、confirm |
| app-ai | AI 對話、Groq client、tool enrichment、report |
| app-rag | RAG document flow、AI Worker integration |
| app-files | 檔案上傳、local/S3/MinIO abstraction |
| app-persistence | Entity、Repository、Flyway migration |
| app-common | ErrorCode、Result、Trace、Enum、共用抽象 |

為什麼不用單一 module：

- 單一 module 初期簡單，但功能多後容易變成大型 service 層互相依賴。
- 本專案有 auth、portfolio、stocks、ocr、ai、rag、files 多個 bounded context，拆 module 可以控制依賴方向。

為什麼不用微服務：

- 微服務會增加部署、觀測、資料一致性與服務間通訊成本。
- 目前團隊與功能階段更適合 modular monolith：程式內部清楚分模組，但部署仍是一個後端服務。

面試講法：

> 我沒有一開始就拆微服務，而是採用 modular monolith。原因是這個階段最重要的是維持交易資料一致性與開發效率。透過 Maven module 控制 auth、portfolio、stocks、ocr、ai 等邊界，未來如果某個模組壓力變大，也比較容易抽出去。

### Ports and Adapters / Hexagonal 思路

專案中比較明顯的抽象：

| Port | Adapter / Implementation |
|---|---|
| StockMarketClient | FugleClient、AlpacaClient、TPEx/TWSE client |
| AI Worker Client | OCR / RAG HTTP client |
| File Storage | local、S3、MinIO |
| CurrentUserProvider | 隔離 Spring Security 細節 |

為什麼這樣設計：

- 第三方股票 API 可能會換供應商。
- OCR / Embedding provider 可能會從 OpenAI 換 Gemini、Ollama 或 Tesseract。
- 檔案儲存可能從 local 換到 S3/MinIO。

面試講法：

> 我把容易變動的外部服務包成 Port/Adapter，例如行情來源、AI Worker、檔案儲存。這樣 domain service 不需要知道是 Fugle 還是 Alpaca，也不需要知道檔案存在 local 還是 S3，之後替換供應商時影響範圍比較小。

## 4. 資料庫與資料設計

### PostgreSQL

| 選型 | 為什麼使用 |
|---|---|
| PostgreSQL | 關聯資料強、交易一致性好、支援 JSONB、支援 pgvector |

為什麼不用 MySQL：

- MySQL 可以處理一般關聯資料，但 PostgreSQL 在 JSONB、複雜查詢、extension 生態與 pgvector 支援上更適合本專案。
- 本專案同時有交易資料、OCR JSON、RAG 向量資料，PostgreSQL 可以集中處理。

為什麼不用 MongoDB：

- OCR 結果與 AI output 可以是 JSON，但核心交易、持倉、使用者、商品主檔是高度關聯資料。
- 若用 MongoDB，交易一致性、join、查詢約束與資料完整性會更難維護。
- PostgreSQL JSONB 已足夠處理半結構化資料。

面試講法：

> 我選 PostgreSQL 是因為這個系統核心是交易與持倉，資料關聯性很強，需要 transaction、foreign key、index 與一致性。同時 OCR 和 AI 結果需要彈性欄位，所以用 JSONB。RAG 則用 pgvector，避免再額外維護一套向量資料庫。

### JSONB

使用場景：

| 表 | 用途 |
|---|---|
| statements.parsed_json | 儲存 OCR 原始解析結果 |
| statement_trades.errors_json / warnings_json | 儲存草稿驗證錯誤與警告 |
| statement_trades.parsed_a_json / parsed_b_json / final_json / diff_json | 支援 OCR 雙路徑解析與比對 |
| ai_reports.input_summary | 儲存 AI 分析輸入摘要 |
| rag_documents.meta / rag_chunks.meta | 儲存 RAG metadata |

為什麼不用全部正規化成欄位：

- OCR 與 AI 結果格式會演進，若每次都改欄位會增加 migration 成本。
- JSONB 可以保存原始結果與 trace，正式交易欄位仍維持結構化。

面試講法：

> 我不是把所有資料都丟 JSON，而是把穩定的交易欄位正規化，把會變動的 OCR/AI metadata 放 JSONB。這樣兼顧資料一致性與彈性。

### pgvector

使用場景：

- `vector.rag_documents`
- `vector.rag_chunks`
- 儲存 chunk embedding
- RAG query 時依 `user_id` 過濾，避免跨使用者資料洩漏

為什麼不用 Pinecone / Weaviate / Qdrant：

- 專案初期資料量還沒大到需要獨立向量資料庫。
- PostgreSQL + pgvector 可以降低系統複雜度與部署成本。
- 向量資料與使用者權限、文件 metadata 在同一個 DB 內，資料隔離更直覺。

何時會考慮換獨立向量 DB：

- 向量資料量很大。
- 查詢延遲成為主要瓶頸。
- 需要更完整的 distributed vector search 能力。

面試講法：

> RAG 初期我選 pgvector，是因為可以少維護一套基礎設施，並且跟使用者資料、文件 metadata 放在同一個 PostgreSQL 裡，權限過濾比較直觀。若未來資料量或查詢延遲成為瓶頸，再評估 Qdrant 或 Pinecone。

### Flyway

為什麼使用：

- DB schema 需要版本控管。
- 多人協作時需要知道每次 schema 變更的順序與原因。
- 測試與部署環境能套用相同 migration。

為什麼不用 Hibernate ddl-auto：

- `ddl-auto` 不適合正式環境，容易產生不可控 schema 變更。
- 金融/交易類資料需要可追溯 migration。

面試講法：

> 我用 Flyway 管 schema，而不是讓 Hibernate 自動改表，因為交易系統的資料庫變更必須可追溯、可審查、可重跑。Hibernate 只負責 mapping，不負責演進 schema。

## 5. Redis 使用場景

### Redis Cache

使用場景：

| Key | 用途 |
|---|---|
| `quote:{symbolKey}` | 即時報價快取 |
| `candles:{symbolKey}:{interval}:{from}:{to}` | K 線資料快取 |

為什麼要快取行情：

- 第三方行情 API 有費用與 rate limit。
- 報價短時間內大量重複查詢，適合短 TTL cache。
- 降低外部 API 延遲對使用者體驗的影響。

為什麼不用只靠 DB cache：

- 行情是高頻查詢與短效資料，Redis 比 DB 更適合。
- 可以搭配 TTL 控制資料新鮮度。

面試講法：

> 股票報價很容易被多個使用者同時查同一檔，如果每次都打外部 API，成本跟 rate limit 都會很快成為問題。所以我用 Redis 做短 TTL 快取，並且在後端做 single flight 防止快取失效瞬間打爆供應商。

### Redis Session / Refresh Token

使用場景：

- `sess:refresh:{jti}`
- Refresh token rotation
- 登出與撤銷 session

為什麼 JWT 還需要 Redis：

- Access token 可以 stateless，但 refresh token 需要能撤銷與防重放。
- 使用者登出後，後端要能讓 refresh session 失效。

面試講法：

> Access token 驗證可以 stateless，但 refresh token 如果完全 stateless，登出和盜用偵測會很弱。所以我把 refresh token 的 jti 存 Redis，每次 refresh 都旋轉，舊 jti 立即失效。

### Redis Streams

使用場景：

- OCR job queue
- 非同步處理 OCR 任務
- 支援 consumer group、retry、dedupe

為什麼不用直接同步呼叫 OCR：

- OCR/LLM 是長任務，直接同步容易造成 HTTP timeout。
- 多個 OCR 任務需要排隊與背景處理。

為什麼不用 Kafka：

- Kafka 適合大型事件流平台，但部署與維運成本較高。
- 本專案 OCR queue 初期需求較單純，Redis Streams 足夠。

面試講法：

> OCR 是長任務，所以我用 Redis Streams 做 queue，讓 API 可以快速建立 job 並回傳，背景 worker 再慢慢處理。沒有一開始用 Kafka，是因為目前事件量與需求還不到那個等級，用 Redis 可以降低維運成本。

## 6. 認證與安全設計

### HttpOnly Cookie + JWT

| 選型 | 原因 |
|---|---|
| JWT Access Token | 短效、API 驗證成本低 |
| Refresh Token | 長效登入狀態 |
| HttpOnly Cookie | 防止 token 被 JavaScript 讀取，降低 XSS token 竊取風險 |
| Redis Refresh Session | 支援撤銷、rotation、防重放 |

為什麼不用 localStorage 存 token：

- localStorage 可被 JavaScript 存取，XSS 時 token 容易被偷。
- HttpOnly Cookie 無法被前端 JS 讀取，安全性較好。

為什麼不用傳統 server session：

- 傳統 session 也可行，但 access token 使用 JWT 可以降低每次 API 查 session 的成本。
- Redis 只管理 refresh session，兼顧 stateless access 與可撤銷 refresh。

面試講法：

> 我沒有把 token 放 localStorage，而是用 HttpOnly Cookie，主要是降低 XSS 竊取 token 的風險。Access token 短效，Refresh token 放 Redis 管 jti，每次 refresh 都 rotation，這樣可以兼顧 API 驗證效率與後端撤銷能力。

### Admin Login + Argon2id

為什麼使用 Argon2id：

- Argon2id 是現代密碼雜湊演算法，抗 GPU 暴力破解能力比傳統 hash 更好。
- 可調 memory、iteration、parallelism。

為什麼不用 SHA256：

- SHA256 是一般 hash，不適合密碼儲存，速度太快，容易被暴力破解。

為什麼不用只靠 Google OAuth：

- 一般使用者用 Google OAuth 降低註冊成本。
- 管理員需要本地帳密入口，避免 OAuth 設定異常時完全無法進入管理端。

面試講法：

> 一般使用者走 Google OAuth，管理員另外提供本地登入。管理員密碼用 Argon2id，不用 SHA256，因為密碼 hash 需要刻意慢且可調 memory cost，才比較能抵抗暴力破解。

### CORS / CSRF

重點：

- CORS 使用白名單 origin。
- Cookie-based auth 需要 `allowCredentials=true`。
- 生產環境不能使用 `*` 搭配 credentials。
- CSRF 可依環境開關，啟用時走 `XSRF-TOKEN` cookie + `X-XSRF-TOKEN` header。

面試講法：

> 因為系統用 Cookie 驗證，所以 CORS 不能隨便開 `*`，必須限制 allowed origins。CSRF 則提供開關，dev 可以降低開發摩擦，prod 可以啟用 token 檢查。

## 7. 股票行情代理設計

### 後端代理第三方行情 API

資料來源：

| 市場 | Adapter |
|---|---|
| 台股 | FugleClient |
| 美股 | AlpacaClient |
| 權證 | TPEx / TWSE fallback |

為什麼前端不直接打第三方 API：

- API key 不能暴露在瀏覽器。
- 第三方回傳格式不同，需要後端正規化。
- 後端可以做 cache、rate limit、single flight 與 observability。

面試講法：

> 前端不直接打 Fugle 或 Alpaca，因為 API key 不能暴露，而且不同供應商格式不一致。我讓後端做 proxy，統一 quote/candle 格式，再加 Redis cache、rate limit 與 single flight。

### Instrument Master / symbol_key

設計：

- 內部主鍵：`instrument_id`，BIGINT 自增。
- 對外唯一鍵：`symbol_key = {market}:{exchange}:{ticker}`。
- 範例：`US:XNAS:AAPL`、`TW:XTAI:2330`。
- 支援 alias，例如 `BRK.B` / `BRK-B`。

為什麼不用 ticker 當唯一鍵：

- 不同市場可能有相同 ticker。
- 同一檔商品在不同來源可能格式不同。
- 權證、ETF、股票都需要統一識別。

為什麼不用 UUID 當主鍵：

- BIGINT 索引較小，join 效能好。
- 自增 ID 易讀、方便 debug。
- 本系統單一主資料庫，不需要 UUID 的全域唯一性。

面試講法：

> 股票不能只用 ticker 當唯一鍵，因為台股、美股、ETF、權證都可能有代號衝突或格式差異。所以我設計了 `symbol_key` 作為跨市場唯一識別，內部仍用 BIGINT `instrument_id` 做 join，兼顧可讀性與效能。

## 8. OCR 匯入設計

### Draft -> Review -> Confirm

流程：

```text
Upload -> files
OCR Job -> ocr_jobs
Parse -> statements / statement_trades
Review -> 使用者修正草稿
Confirm -> stock_trades + user_positions
```

為什麼 OCR 結果不直接入帳：

- OCR 可能辨識錯日期、數量、價格或股票代號。
- 投資帳務不能讓 AI/OCR 自動寫入正式交易。
- Review 階段可以保留 warnings/errors，讓使用者修正。

面試講法：

> OCR 結果一定先進草稿，不直接入帳。因為帳務資料不能完全相信 OCR 或 LLM，必須讓使用者 Review 後 Confirm，才寫入正式交易與更新持倉。

### Validator Chain

驗證內容：

- 同一 statement 內重複 row hash。
- 必填欄位缺失。
- 交割日早於交易日。
- 交易日期在未來。
- 幣別不一致。
- 與既有正式交易重複。

為什麼用 Validator Chain：

- 驗證規則會持續增加。
- 若全部寫在 `saveDrafts()`，service 會快速膨脹。
- 每個 validator 單一職責，方便測試與維護。

面試講法：

> OCR 草稿驗證我用 Validator Chain，因為規則會一直增加，例如必填欄位、重複交易、日期異常、幣別不一致。拆成 chain 後每條規則都能獨立測試，不會讓核心 service 越來越大。

### Python AI Worker

為什麼 OCR/RAG 另外拆 Python：

- Python AI 生態成熟，OpenAI、Tesseract、PDF parsing、Embedding、向量處理都方便。
- OCR 和文件 ingestion 是 CPU/I/O/LLM 長任務，不適合塞在 Java API request thread。
- Java 後端維持核心業務穩定，Python Worker 專注 AI 任務。

為什麼不用 Java 直接做全部 OCR：

- Java 能做，但 AI/OCR provider、PDF/image processing、embedding 生態不如 Python 彈性。
- 會讓後端主服務依賴更多 AI/影像處理套件，增加複雜度。

面試講法：

> 我把 AI Worker 拆成 Python FastAPI，因為 OCR、PDF parsing、embedding 這些任務在 Python 生態比較成熟。Java 後端只負責建立 job、控管流程和寫正式資料，Worker 負責重型 AI 處理。

## 9. AI / RAG 設計

### SSE 串流

為什麼使用 SSE：

- AI 回覆是 server-to-client 單向串流。
- SSE 基於 HTTP，前端實作簡單，Nginx 也容易支援。
- 適合文字逐段輸出。

為什麼不用 WebSocket：

- WebSocket 適合雙向即時互動、多人協作、即時推播。
- AI 回覆目前主要是單向 streaming，用 WebSocket 會增加握手、心跳、重連與權限處理成本。

面試講法：

> AI 回覆是後端往前端單向串流，所以我選 SSE。WebSocket 更適合雙向聊天室或盤中大量推播，目前會增加不必要的複雜度。

### Groq / OpenAI Compatible Client

使用方式：

- 後端 AI 對話使用 Groq OpenAI-compatible API。
- 模型與參數透過環境變數設定。
- 後端在送 LLM 前先做 tool enrichment，例如 instrument candidates、quote、portfolio context、RAG context。

為什麼不讓 LLM 直接呼叫外部 API：

- 權限與資料隔離要由後端控制。
- 報價、持倉與文件檢索需要使用者身份。
- 後端可記錄 tool result 與錯誤，方便 debug。

面試講法：

> 我沒有讓 LLM 自己亂呼叫工具，而是由後端先根據使用者訊息查標的、報價、持倉與 RAG，再把結果注入 prompt。這樣權限、安全與資料正確性都由後端控制。

### RAG

流程：

```text
File / Text -> Parse -> Chunk -> Embedding -> pgvector
Query -> Embedding -> TopK retrieval -> citations -> LLM answer
```

重點：

- 每個 document/chunk 都帶 `user_id`。
- 查詢時必須用 `user_id` 過濾。
- `ai-worker` 負責 row mutation，schema 由 backend Flyway 管理。
- 啟動時檢查 embedding dimension 是否與 DB `vector(n)` 一致。

為什麼重視 dimension guard：

- 不同 embedding model 維度不同，例如 1536、3072、768。
- DB vector 欄位維度不一致時，寫入或查詢會失敗。
- 啟動時 fail-fast 比 runtime 才爆錯更容易排查。

面試講法：

> RAG 我特別處理 embedding dimension guard，因為不同模型維度不同。如果 DB 是 `vector(1536)`，但 worker 換成 768 維模型，系統會在啟動時檢查出來，而不是等使用者查詢時才失敗。

## 10. 投資組合與交易計算

### 加權平均成本

規則：

- BUY：更新總數量與平均成本。
- SELL：減少數量，平均成本不變。
- 異常時可透過 positions rebuild 重新從交易明細回算。

為什麼不用 FIFO：

- FIFO 對稅務或精準已實現損益有用，但 MVP 複雜度較高。
- 加權平均成本更直觀，適合先滿足持倉成本與未實現損益。

為什麼需要 positions 快照表：

- 每次查持倉若都從所有 trades 即時計算，成本較高。
- `user_positions` 是衍生快照，查詢快。
- 若資料異常，可用 rebuild 從 `stock_trades` 重算。

面試講法：

> 交易明細是 source of truth，`user_positions` 是查詢用快照。這樣查持倉很快，真的出現異常時可以從交易明細 rebuild。成本法先用加權平均，因為它符合 MVP 需求且使用者容易理解。

| postgres | PostgreSQL + pgvector |

## 15. 面試常見問答

### Q1：這個專案最大的技術挑戰是什麼？

可以回答：

> 最大挑戰是把股票交易這種需要資料一致性的業務，跟 OCR/AI 這種不確定性高的功能整合在一起。我用 Draft -> Review -> Confirm 的流程隔離風險，讓 OCR 和 LLM 只產生草稿，正式交易一定要使用者確認後才入帳。

### Q2：為什麼後端不用微服務？

可以回答：

> 目前業務還在快速演進，如果太早拆微服務，會增加部署、觀測、資料一致性與服務間通訊成本。我選 modular monolith，先把程式邊界切清楚，未來如果 OCR、行情或 AI 模組流量變大，再把它們獨立出去。

### Q3：為什麼 RAG 不用獨立向量資料庫？

可以回答：

> 初期資料量還不大，而且每筆向量都需要跟 user、document metadata 做權限過濾。pgvector 可以直接放在 PostgreSQL 裡，降低系統複雜度。等資料量或查詢效能成為瓶頸，再評估 Qdrant、Weaviate 或 Pinecone。

### Q4：為什麼 token 放 Cookie，不放 localStorage？

可以回答：

> localStorage 會被 JavaScript 存取，XSS 時 token 容易被偷。HttpOnly Cookie 可以降低 token 被前端 JS 竊取的風險。Refresh token 還會用 Redis 存 jti 並做 rotation，所以可以撤銷與防重放。

### Q5：為什麼 AI 回覆用 SSE，不用 WebSocket？

可以回答：

> 目前 AI 回覆是後端往前端單向傳文字，SSE 很適合這個場景，基於 HTTP、實作簡單、反向代理也容易處理。WebSocket 更適合雙向即時互動，例如多人聊天室或盤中推播，目前用它會增加不必要的複雜度。

### Q6：怎麼避免第三方股票 API 被大量請求打爆？

可以回答：

> 我做了 Redis cache、rate limit，以及 single flight。Redis 降低重複查詢，rate limit 控制外部 API 呼叫頻率，single flight 則避免同一檔股票快取剛過期時，多個請求同時打到供應商。

### Q7：OCR 匯入如何確保資料正確？

可以回答：

> OCR 解析後只會寫入 statement_trades 草稿表，不會直接寫正式交易。草稿會經過 Validator Chain 檢查必填欄位、日期異常、重複交易、幣別不一致等問題，最後由使用者在 Review 頁確認後才寫入 stock_trades 並更新 user_positions。

### Q8：這個專案怎麼處理台股和美股代號衝突？

可以回答：

> 我設計了 instrument master。內部用 BIGINT `instrument_id` 做主鍵，對外用 `symbol_key = market:exchange:ticker` 做唯一識別，例如 `US:XNAS:AAPL`、`TW:XTAI:2330`。這樣可以避免只用 ticker 造成市場衝突。

### Q9：如果要擴充到更多行情供應商，要怎麼做？

可以回答：

> 行情來源透過 `StockMarketClient` port 抽象，Fugle、Alpaca、TPEx 都是 adapter。新增供應商時主要是新增 adapter，轉成內部統一的 Quote/Candle model，不需要大改 controller 或 portfolio service。

### Q10：這個專案有哪些可以再優化的地方？

可以回答：

> 可以優化的方向包括：把 in-memory rate limiter 改成 Redis-based 以支援多實例；RAG 資料量變大後評估 HNSW 或獨立向量 DB；OCR Worker 可以進一步做更完整的 retry、dead-letter queue；前端測試覆蓋率也可以補到主要工作流，例如 OCR Review 和 Chat SSE。
