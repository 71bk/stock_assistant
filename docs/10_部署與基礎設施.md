# 部署與基礎設施

> 狀態：已整理可落地版本 (2026-02-09)

## Docker Compose

- 檔案位置：`apps/backend/invest-assistant-backend/compose.yaml`
- 目前內建服務：
  - `postgres`：`pgvector/pgvector:pg16`
  - `redis`：`redis:7.2`（AOF 開啟）
- 啟動：
  - `cd apps/backend/invest-assistant-backend`
  - `docker compose up -d`
- 停止：
  - `docker compose down`

## Nginx 反向代理（含 SSE）

- 建議反代 `/api` 到 Spring Boot（context path 已是 `/api`）。
- SSE 必要設定：
  - `proxy_buffering off`
  - `proxy_read_timeout` 調大（例如 300s）
  - `chunked_transfer_encoding on`

## CI/CD（GitHub Actions）

- 建議最小流程：
  - `mvn -pl modules/app-api -am test-compile`
  - 架構規則測試（ArchUnit）
  - `mvn -pl modules/app-stocks -am test`
  - 建置 image 並推送 registry
  - 目標機器 `docker compose pull && docker compose up -d`
- 生產環境若要上傳 Sentry source bundle，CI 要注入 `SENTRY_AUTH_TOKEN`。

## 測試與驗證

- RAG schema ownership：
  - `vector.rag_documents`、`vector.rag_chunks` 由 backend Flyway migration 定義。
  - `ai-worker` 依既有 schema 寫入與查詢，並擁有這兩張表的 row mutation，不單獨演化資料表結構。
  - 規則詳見 `docs/adr/0004_rag_schema_ownership.md`。
- Backend schema contract test：
  - `cd apps/backend/invest-assistant-backend`
  - `mvn -pl modules/app-persistence -am test "-Dtest=RagVectorSchemaContractTest" "-Dsurefire.failIfNoSpecifiedTests=false"`
  - 此測試使用 Testcontainers 啟動 `pgvector/pgvector:pg16`，執行前需要本機可用 Docker environment。
- AI Worker schema contract test：
  - `cd apps/ai-worker`
  - `python -m unittest tests.test_rag_schema_contract`
  - 這支測試只檢查 worker 端的函式簽名與 SQL 契約，不依賴資料庫。
- AI Worker embedding dimension guard：
  - `ai-worker` 啟動時會先驗證 provider/model 對應的預期 embedding dimension。
  - `init_pool()` 後會再驗證 `vector.rag_chunks.embedding` 的實際 DB dimension 是否等於 `embedding_dimension`。
  - 若未來切換到 `text-embedding-3-large`、`nomic-embed-text` 等不同維度模型，必須先同步調整 Flyway schema。

## 監控與備份

- Metrics/Monitoring：
  - Production Compose 會啟動內網 Prometheus，抓取 `backend:8080/api/actuator/prometheus`
  - 同時抓取 `ai-worker:8001/metrics` 的 AI/OCR/Embedding 用量
  - Prometheus 不發布 host port，公開 Nginx 會封鎖 `/api/actuator/prometheus`
  - TSDB 預設保留 30 天，可用 `.env.prod` 的 `PROMETHEUS_RETENTION_TIME` 調整
  - 監控重點：`stock.external.*`、`stock.cache.requests`、`stock.singleflight.joined`
- Logs：
  - 保留 `traceId` 以便 API 請求追蹤
  - 外部行情呼叫保留 `vendor/endpoint/status/latency_ms/request_id/trace_id`
- 備份（已落地，`db-backup` service，原始碼於 `infra/backup/`）：
  - PostgreSQL：每日 `pg_dump`（custom format）→ 本地 volume `backup_data` 保留 7 天 → 上傳 Cloudflare R2 保留 30 天
  - 在 host 本機從 `infra/backup` build；R2 為選用，未設 `R2_*` 變數時僅做本地備份
  - 相關變數見 `.env.prod.example` 的「Database backups」段（`BACKUP_KEEP_DAYS` / `R2_KEEP_DAYS` / `R2_BUCKET` / `R2_ENDPOINT` / `R2_ACCESS_KEY_ID` / `R2_SECRET_ACCESS_KEY`）
  - 手動立即備份一次：`docker compose -f compose.prod.yaml run --rm db-backup once`
  - 還原（**具破壞性**，會先 drop 既有物件）：
    - 本地檔：`docker compose -f compose.prod.yaml run --rm db-backup restore /backups/<檔名>.dump`
    - 從 R2 取回再還原：先 `... run --rm db-backup sh -c 'rclone copy "R2:$R2_BUCKET/$R2_PREFIX/<檔名>.dump" /backups/'`，再執行上面的 restore
  - Redis：已開啟 AOF（`--appendonly yes`）持久化；如含重要 session 狀態可另加定期快照
