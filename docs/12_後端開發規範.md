# 後端開發規範

> 狀態：已實作 (2026-02-09)

## 分層與命名

遵循 **Maven Multi-Module** 結構與 **Spring Boot** 標準分層：

- **Controller Layer** (`web` / `api` package):
  - 處理 HTTP 請求、參數驗證 (`@Valid`)。
  - **禁止**在此層寫入複雜業務邏輯或 SQL。
  - 回傳型別統一為 `Result<T>` 或 `ResponseEntity<Result<T>>`。

- **Service Layer** (`service` package):
  - 核心業務邏輯。
  - 交易邊界 (`@Transactional`) 所在。
  - 負責 Entity 與 DTO 之間的轉換。

- **Repository Layer** (`repository` package):
  - 資料存取層 (Spring Data JPA)。
  - 方法命名遵循 JPA 規範 (e.g., `findByTicketAndDate`)。

- **Entity Layer** (`entity` package):
  - 對應資料庫表結構。
  - 使用 Lombok (`@Data`, `@Builder`) 簡化程式碼。

## DTO/VO/BO

為了避免將 Entity 直接暴露給前端，需使用不同的資料傳輸物件：

- **Request Object**: 用於 Controller 接收參數 (e.g., `LoginRequest`)。
- **Response Object / VO**: 用於 Controller 回傳資料 (e.g., `LoginResponse`)。
- **DTO (Data Transfer Object)**: 模組間或服務間的資料傳遞。

## Exception 與 ErrorCode

統一的例外處理機制：

1. **定義錯誤碼**: 在 `ErrorCode` enum 中定義所有已知錯誤 (如 `AUTH_UNAUTHORIZED`, `USER_NOT_FOUND`)。
2. **拋出異常**: 業務邏輯中遇到錯誤，直接拋出 `BusinessException`。
   ```java
   throw new BusinessException(ErrorCode.USER_NOT_FOUND, "User ID not found: " + id);
   ```
3. **全域捕捉**: `GlobalExceptionHandler` 會自動捕捉並轉換為標準 JSON 回傳。

## Controller 共用工具與 ID 解析

Controller 共用的 web 工具集中在 `tw.bk.appapi.web`，避免各 controller 重複實作：

- `CurrentUser.require(currentUserProvider)`：取得當前登入 `userId`，未登入丟 `AUTH_UNAUTHORIZED`。
- `IdParser.parseId / parseIdOrNull`：解析**字串來源**的數字 ID（path 以外的 body / list / query），格式錯誤丟 `VALIDATION_ERROR`。
- `PageableFactory.of(page, size, maxSize, sort)`：統一 1-based→0-based 分頁正規化與邊界處理。

ID 路徑參數慣例：

- **數字型 ID 的 path 參數一律以 `@PathVariable String` 接收，再用 `IdParser.parseId` 轉 `Long`**（非數字鍵如 `symbolKey` 維持 String 不解析）。
- **暫不採用 typed `@PathVariable Long`**：`GlobalExceptionHandler` 目前未處理 `MethodArgumentTypeMismatchException`。若直接改 typed，path 格式錯誤（如 `/portfolios/abc`）會在 Spring 綁定階段丟 `MethodArgumentTypeMismatchException`，掉到通用 handler 變成 **500**，而非現在的 400 `VALIDATION_ERROR`。
- 日後若要改用 typed `@PathVariable Long`，順序為：先在 `GlobalExceptionHandler` 補 `MethodArgumentTypeMismatchException → VALIDATION_ERROR(400)` 的 handler（順帶把錯誤的 query 參數從 500 修成 400），再逐一轉換 controller 與對應單元測試（測試直接呼叫 controller 方法、目前傳字串 ID，需一併改為 `Long`）。

## 交易邊界（@Transactional）

- 預設在 **Service Class** 層級加上 ` @Transactional(readOnly = true)`。
- 涉及寫入 (Insert/Update/Delete) 的 Open Public Method 需明確標註 `@Transactional`。
- 跨模組呼叫時，盡量保持交易的傳播行為 (Propagation.REQUIRED)。

## AOP/Interceptor 使用準則

- **Logging**: 可使用 AOP 記錄 Controller 的進入與離開時間 (耗時統計)。
- **Auth**: 使用 `Filter` (如 `JwtAuthenticationFilter`) 處理身分驗證，而非 AOP。
- **Rate Limit**: 可使用 Interceptor 或 Filter 實作限流。

## 架構防線（Build 失敗機制）

- Root `pom.xml` 已啟用 `maven-enforcer-plugin`（`validate` 階段）：
  - Java / Maven 版本檢查
  - 重複依賴版本防呆
  - Reactor module 收斂
- ArchUnit 規則（測試必跑）：
  - `ApiArchitectureRulesTest`：`app-api` 不可依賴 `app-persistence.entity`
  - `CommonArchitectureRulesTest`：`app-common` 不可依賴任何業務模組
- 新增模組或調整依賴時，需同步新增/更新對應 ArchUnit 規則。

## 可觀測性最小標準

- 每個 API 回應必須可追蹤 `traceId`（來自 `TraceIdFilter` / MDC）。
- 外部供應商呼叫 log 至少包含：
  - `vendor`, `endpoint`, `status`, `latency_ms`, `request_id`, `trace_id`
- 新增快取或限流邏輯時，需補對應 metrics（Counter/Timer）以便監控。

## Code Style

- **Java**: 遵循 Google Java Style Guide。
- **Lombok**: 廣泛使用 `@RequiredArgsConstructor` 進行 Constructor Injection (取代 `@Autowired`)。
- **Null Safety**: 優先使用 `Optional<T>` 處理可能為空的回傳值。
